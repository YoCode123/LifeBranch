# LifeBranch

人生の分岐点での意思決定を記録し、未来の自分が振り返るためのアプリ

---

## 1. サービス概要

LifeBranchは、人生の分岐点で行った意思決定と、そのときの思考や感情を記録するアプリです。
記録を振り返ることで、未来の意思決定に迷いが生じたときに参考にでき、次の挑戦を後押しします。

---

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

- プログラミングスクール入学前にキャリア選択で迷った
  - 今の仕事を続ける
  - 独学でプログラミングを学ぶ
  - スクールに通う
  - 別の分野に挑戦する
- 当時の考えや気持ちは時間とともに忘れてしまった
- エラーが解決できず長時間停滞、周囲と比較して焦り、不安を感じた

### 2-2. なぜそれが「気になった」のか

- 過去の迷いや意思決定を振り返ることで、挫折時にも前に進む力になると感じた
- 記録を残すことで、未来の意思決定をより納得感のあるものにできるのではないかと考えた

---

## 3. 課題の整理

### 3-1. 表に見えている困りごと

- 過去の決断理由や迷いを忘れてしまう
- 将来、似たような決断をするときに参考にできない
- 「なぜこの道を選んだのか」を思い出せず不安になる

### 3-2. 本当に解決したい課題

- 自分の意思決定プロセスを記録・振り返り、未来の迷いを減らす
- 過去の判断基準や価値観を参照できる状態にする

---

## 4. 想定ユーザー

### 4-1. 想定しているユーザー

- 20〜30代社会人
- キャリアや挑戦の意思決定に関心がある
- 過去の選択を振り返り、自己成長を可視化したい

### 4-2. 自分とユーザーの距離

- 最初は自身の体験として使用
- 徐々に同じような悩みを持つ他ユーザーへ展開

### 4-3. 実際に使われる可能性

- 人生の節目や迷いのタイミング（課題が難しい、転職活動中）に開く可能性
- 定期的な振り返り習慣も検討可能

### 4-4. 利用イメージ

1. ユーザー登録・ログイン
2. 過去の意思決定を記録
3. 記録内容を一覧で確認
4. 過去の決断パターンを振り返る
5. 次の挑戦や意思決定に活かす

---

## 5. 競合・既存サービスとの比較

| サービス名 | 目的 | 内容 | 頻度 |
|------------|------|------|------|
| Day One | 日常記録 | 出来事・感情 | 毎日 |
| Notion | メモ管理 | テキスト・情報整理 | 毎日 |
| LifeBranch | 人生の意思決定記録 | 選択・理由・迷い・期待 | 人生の節目 |

### 5-2. 自分が作りたい理由

- 人生の意思決定に特化した記録・振り返り体験は既存サービスにない
- 過去の決断を未来の意思決定に活かすことに特化している

### 5-3. 差別化ポイント

> 「人生の分岐点での意思決定を可視化し、未来の迷いを減らす」

---

## 6. 提供する価値

### 6-1. ユーザーの変化

- 過去の選択肢や決断理由が明確になり、「なぜ前回この道を選んだのか」が理解できる
- 同じ意思決定をする際、過去の判断基準や価値観を参考にできる
- 不安な時に「過去の自分も迷ったけど進めた」と思い出すことで迷いを減らす

### 6-2. 価値の一文

> 「過去の意思決定を振り返り、未来の迷いを減らす」

---

## 7. アプリで実現すること

### MVPで作る機能

- ユーザー登録・ログイン(Devise)
- 決断の記録(CRUD)
  - 新規作成
  - 一覧表示
  - 詳細表示
  - 編集
  - 削除
- 記録内容
  - タイトル / 日付 / カテゴリ（選択式、自由入力は今後追加予定）
  - 選択肢 / 最終決断 / 決断理由 / 迷い / 感情（不安・期待・ワクワク）

### 本リリースで作る機能

- タイムライン表示
- 振り返りリマインド(通知)
- 感情の可視化
- 選択パターン分析

## 8. 懸念点と具体的対策

### 懸念点① 過去1〜2年の意思決定を思い出して記録してくれるか

- **対策:**
  - 記録入力のハードルを下げるため、MVPでは「過去の意思決定一覧」を時系列で見返す疑似体験を提供
  - カテゴリはあらかじめ用意された選択式
  - 記録を小分けに入力できるUIにして負担を軽減

### 懸念点② 挫折しそうなタイミングで本当に開くか

- **対策:**
  - MVPでは一覧を見返す疑似体験で振り返りの価値を検証
  - 週1回程度の定期振り返りや、迷った時に一覧を確認する流れを想定
  - 実際の挫折時の使用は本リリースでユーザーテストを通じて検証

## 10. 技術スタック

### 10-1. 使用技術

- Backend / Frontend: Ruby / Ruby on Rails
- Database: PostgreSQL
- 認証: Devise
- フロントエンド: HTML / CSS / Bootstrap
- インフラ: Render
- Docker: 開発・本番環境の統一と依存関係管理
- バージョン管理: Git / GitHub

### 10-2. 技術選定の理由

- Rails: CRUDと認証機能を素早く作れる
- Devise: セキュアなユーザー認証を簡単に実装
- PostgreSQL / SQLite: データ管理と拡張性
- HTML/CSS/Bootstrap: シンプルでレスポンシブなUIを迅速に作れる
- Render: デプロイと運用が簡単
- Docker: 環境差異をなくし、チーム開発やCI/CDが容易になる
