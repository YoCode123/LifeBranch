# LifeBranch

人生の分岐点での意思決定を記録し、未来の自分が振り返るためのアプリ

---

# 1. サービス概要

LifeBranchは、人生の分岐点で行った意思決定と、そのときの思考や感情を記録するアプリです。  
記録を振り返ることで、未来の意思決定に迷いが生じたときに参考にでき、次の挑戦を後押しします。

本サービスは、日常的に記録する日記アプリというよりも  
**人生の節目や挑戦のタイミングで振り返る意思決定ログツール**としての利用を想定しています。

---

# 2. このアイデアはどこから生まれたか

## 2-1. きっかけとなった体験・感情

- プログラミングスクール入学前にキャリア選択で迷った
  - 今の仕事を続ける
  - 独学でプログラミングを学ぶ
  - スクールに通う
  - 別の分野に挑戦する
- 当時の考えや気持ちは時間とともに忘れてしまった
- エラーが解決できず長時間停滞、周囲と比較して焦り、不安を感じた

## 2-2. なぜそれが「気になった」のか

- 過去の迷いや意思決定を振り返ることで、挫折時にも前に進む力になると感じた
- 記録を残すことで、未来の意思決定をより納得感のあるものにできるのではないかと考えた

---

# 3. 課題の整理

## 3-1. 表に見えている困りごと

- 過去の決断理由や迷いを忘れてしまう
- 将来、似たような決断をするときに参考にできない
- 「なぜこの道を選んだのか」を思い出せず不安になる

## 3-2. 本当に解決したい課題は何か

- 自分の意思決定プロセスを記録・振り返り、未来の迷いを減らす
- 過去の判断基準や価値観を参照できる状態にする

---

# 4. 想定ユーザーについて

## 4-1. 想定しているユーザー

- 20〜30代社会人
- キャリアや挑戦の意思決定に関心がある
- 過去の選択を振り返り、自己成長を可視化したい

## 4-2. 自分とユーザーの距離

- 最初は自身の体験として使用
- 徐々に同じような悩みを持つ他ユーザーへ展開

## 4-3. 実際に使われる可能性

ユーザーは主に以下のタイミングでサービスを利用すると想定しています。

1. 新しい挑戦を始める前  
2. 挑戦中に不安を感じたとき  
3. 挫折しそうなとき  
4. 挑戦が終わったあと  

特に本サービスでは  
**過去の意思決定を振り返ることで自己効力感を高める体験**を提供します。

## 4-4. ユーザーがサービスを導入して利用するまでのイメージ

1. ユーザー登録・ログイン  
2. 過去の意思決定を記録  
3. 記録内容を一覧で確認  
4. 過去の決断パターンを振り返る  
5. 次の挑戦や意思決定に活かす  

また、初回利用時は記録が存在しないため  
**過去の意思決定を1件入力する導線**を用意します。

例

- 転職を迷った経験
- 新しい挑戦を決めた経験
- 学習を始めた経験

これにより、サービス初期段階でも  
**意思決定の振り返り体験をすぐに得られる状態**を作ります。

# 5. 既存サービス・競合調査

## 5-1. 似たサービスの調査

| サービス名 | 目的 | 内容 | 頻度 |
|------------|------|------|------|
| Day One | 日常記録 | 出来事・感情 | 毎日 |
| Notion | メモ管理 | テキスト・情報整理 | 毎日 |
| LifeBranch | 人生の意思決定記録 | 選択・理由・迷い・期待 | 人生の節目 |

## 5-2. それでも自分が作りたい理由

- 人生の意思決定に特化した記録・振り返り体験は既存サービスにない
- 過去の決断を未来の意思決定に活かすことに特化している

## 5-3. 差別化を一文で決める

> 「人生の分岐点での意思決定を可視化し、未来の迷いを減らす」

# 8. このアプリの懸念点とその対策

## 懸念点①

過去1〜2年の意思決定を思い出して記録してくれるか

### 対策

サービス利用開始時は記録が存在しないため、  
**初回入力を促す導線を設けることで入力ハードルを下げる**

具体的には

- 初回利用時に過去の意思決定を1件入力する簡易フォームを表示
- 直近1〜2年で印象に残っている意思決定を入力してもらう
- 入力後すぐに一覧に表示され、振り返り体験を提供

また入力の負担を減らすため

- カテゴリは選択式
- 記録を小分けに入力できるUI

を想定しています。

---

## 懸念点②

挫折しそうなタイミングで本当にサービスを開くのか

### 対策

MVP段階でも以下の方法で検証を行う予定です。

- ユーザーインタビュー
- 簡易ユーザーテスト
- 利用シナリオのヒアリング

これにより

- 挫折時に過去の意思決定を振り返る価値があるか
- 実際にサービスを開く可能性があるか

を確認し、本リリースの機能改善に活かします。

# 10. 技術スタック（手段としての技術）

## 10-1. 使用予定の技術

- Backend / Frontend: Ruby / Ruby on Rails
- Database: PostgreSQL
- 認証: Devise
- フロントエンド: HTML / CSS / Bootstrap
- インフラ: Render
- Docker: 開発・本番環境の統一
- バージョン管理: Git / GitHub

## 10-2. 技術選定の理由

- Rails: CRUDと認証機能を素早く実装できる
- Devise: セキュアなユーザー認証
- PostgreSQL: 本番環境での安定性と拡張性
- HTML/CSS/Bootstrap: シンプルでレスポンシブなUI
- Render: デプロイと運用が容易
- Docker: 環境差異をなくすため
