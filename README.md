# LifeBranch

人生の分岐点での意思決定を記録し、未来の自分が振り返るためのアプリ

---

## 1. サービス概要

LifeBranchは、人生の分岐点で行った意思決定と、そのときの思考や感情を記録するアプリです。
過去の意思決定や迷いを残すことで、未来の自分が振り返りやすくなり、次の挑戦を後押しします。

---

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

- プログラミングスクール入学前、キャリア選択に迷っていた
  - 今の仕事を続ける
  - 独学でプログラミングを学ぶ
  - プログラミングスクールに通う
  - 別の分野に挑戦する
- 当時の考えや気持ちは時間が経つと忘れてしまった
- エラーが解決できず何時間も止まる、周囲と比較して焦る、選択に不安を感じる

### 2-2. なぜそれが「気になった」のか

- 挫折や迷いの中で、「なぜこの道を選んだのか」を振り返れる体験があれば前に進む力になると感じた
- 過去の意思決定や迷いを残すことで、未来の意思決定に役立てられるのではないかと思った

---

## 3. 課題の整理

### 3-1. 表に見えている困りごと

- 過去の決断理由や迷いを忘れてしまう
- 将来、似たような決断をするときに参考にできない
- 「なぜこの道を選んだのか」を思い出せず不安になる

### 3-2. 本当に解決したい課題は何か

- 自分の意思決定プロセスを記録し、振り返ることで未来の迷いを減らす
- 過去の判断基準や価値観を参照できるようにする

---

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー

- 20代〜30代の社会人
- キャリアや新しい挑戦の意思決定に関心がある
- 過去の選択を振り返りたい、自己成長を可視化したい

### 4-2. 自分とユーザーの距離

- 最初は自分自身が使うことを想定
- 徐々に同じような悩みを持つ他ユーザーにも提供

### 4-3. 実際に使われる可能性について

- 人生の節目や迷いのタイミング（課題が難しい、転職活動中など）に開く可能性
- 定期的な振り返り習慣を作ることも検討可能

### 4-4. ユーザーがサービスを導入して利用するまでのイメージ

1. ユーザー登録・ログイン
2. 過去の意思決定を記録
3. 記録した内容を一覧で確認
4. 過去の決断パターンを振り返る
5. 次の挑戦や意思決定に活かす

---

## 5. 既存サービス・競合調査

### 5-1. 似たサービスの調査

| サービス名 | 目的 | 内容 | 頻度 |
|------------|------|------|------|
| Day One | 日常の記録 | 出来事・感情 | 毎日 |
| Notion | メモ管理 | テキスト・情報整理 | 毎日 |
| LifeBranch | 人生の意思決定記録 | 選択・理由・迷い・期待 | 人生の節目 |

### 5-2. それでも自分が作りたい理由

- 人生の意思決定に特化した記録・振り返り体験は既存サービスにはない
- 過去の決断を未来の意思決定に活かすことに特化している

### 5-3. 差別化を一文で決める

> 「人生の分岐点での意思決定を可視化し、未来の迷いを減らす」

---

## 6. このサービスで提供したい価値

### 6-1. ユーザーの変化

- 過去の選択肢や決断理由を振り返り、「なぜ前回この道を選んだのか」が明確になる
- 同じような意思決定をするときに、過去の自分の価値観や判断基準を参考にできる
- 不安な時に「過去の自分も迷ったけど進めた」と思い出すことで、迷いを減らし行動に移せる

### 6-2. 価値を一文で表す

> 「過去の意思決定を振り返り、未来の迷いを減らす」

## 7. このアプリで実現すること

### MVPで作る機能

- ユーザー登録・ログイン(Devise)
- 決断の記録(CRUD)
  - 新規作成
  - 一覧表示
  - 詳細表示
  - 編集
  - 削除
- 記録内容: タイトル / 日付 / カテゴリ / 選択肢 / 最終決断 / 決断理由 / 迷い / 感情（不安・期待・ワクワク）

### 本リリースで作る機能（今後の拡張）

- タイムライン表示
- 振り返りリマインド(通知)
- 感情の可視化
- 選択パターン分析

## 8. このアプリの懸念点とその対策

### 懸念点①

- 過去1〜2年の意思決定をユーザーが思い出して記録してくれるか
- **対策:** 疑似的に一覧で振り返る体験を提供し、入力のハードルを下げる

### 懸念点②

- 「挫折しそうなタイミング」で本当に開くのか
- **対策:** MVPでは一覧を見返す疑似体験で検証、実際の使用はユーザーテストで確認

## 10. 技術スタック（手段としての技術）

### 10-1. 使用予定の技術

- Backend / Frontend: Ruby / Ruby on Rails
- Database: PostgreSQL（またはSQLite）
- 認証: Devise
- フロントエンド: HTML / CSS / Bootstrap
- インフラ: Render / Heroku（予定）
- バージョン管理: Git / GitHub

### 10-2. 技術選定の理由

- Rails: CRUDと認証機能が素早く作れる
- Devise: セキュアなユーザー認証を簡単に実装可能
- PostgreSQL / SQLite: データ管理と拡張性
- HTML/CSS/Bootstrap: シンプルでレスポンシブなUIを迅速に作れる
- Render / Heroku: デプロイと運用が簡単
